# Eyetrackers Data Model

**Architecture Version:** 1.0  
**Last Updated:** 2026-07-11  
**Status:** Active

---

# Purpose

This document defines the data contracts used throughout the Eyetrackers subsystem.

Every module communicates using these dataclasses. Anonymous dictionaries are no longer used for inter-module communication.

Changing a dataclass described in this document is considered an architectural change and should be reflected throughout the project.

---

# ESP32Metadata

## Owner

MJPEGStream

## Purpose

Represents metadata embedded into every MJPEG frame by the ESP32 firmware together with metadata measured by the host computer.

This object contains timing information only.

---

## Fields

| Field | Type | Description |
|--------|------|-------------|
| frame_number | int | Sequential frame number generated by the ESP32 |
| capture_timestamp_ms | int | Unix timestamp generated by the ESP32 at image capture |
| receive_timestamp_ms | int | Host timestamp when the frame was completely received |
| clock_offset_ms | int | Estimated difference between host receive time and ESP capture time |

---

## Notes

The ESP32 capture timestamp is considered authoritative.

Host timestamps are used only for diagnostics and latency estimation.

---

# FramePacket

## Owner

MJPEGStream

## Purpose

Represents a single decoded image together with its metadata.

Every frame in the pipeline is represented by exactly one FramePacket.

---

## Fields

| Field | Type |
|--------|------|
| image | numpy.ndarray |
| metadata | ESP32Metadata |

---

## Convenience Properties

These properties exist to simplify downstream code.

| Property | Equivalent |
|----------|------------|
| frame_number | metadata.frame_number |
| capture_ms | metadata.capture_timestamp_ms |
| receive_ms | metadata.receive_timestamp_ms |
| latency_ms | receive_ms - capture_ms |

---

## Lifetime

Created once by MJPEGStream.

Never modified afterwards.

---

# FrameBuffer

## Owner

Camera

## Purpose

Provides thread-safe ordered storage of FramePacket objects.

FrameBuffer does not perform synchronization.

---

## Guarantees

FramePackets remain ordered by capture timestamp.

Old frames may be discarded according to configured limits.

Thread-safe access is provided to all callers.

---

# SyncPair

## Owner

StereoSynchronizer

## Purpose

Represents two synchronized FramePackets.

All downstream consumers operate exclusively on SyncPair objects.

---

## Fields

| Field | Type |
|--------|------|
| left | FramePacket |
| right | FramePacket |
| sync_timestamp_ms | int |
| delta_ms | int |

---

## Meaning

sync_timestamp_ms represents the timestamp assigned to the synchronized pair.

delta_ms represents the synchronization error between the two camera frames.

Lower values indicate better synchronization.

# Data Ownership

| Object | Created By | Consumed By |
|---------|------------|-------------|
| ESP32Metadata | MJPEGStream | FramePacket |
| FramePacket | MJPEGStream | Camera |
| FrameBuffer | Camera | StereoSynchronizer |
| SyncPair | StereoSynchronizer | Recorder, CSV Logger, Display |

```
        ▼

        │
        ├──────────────┐
        ▼              ▼
Recorder       CSV Logger
        │
        ▼
Saved Outputs
```

Objects move forward through the pipeline.

Objects should never move backward.

---

# Immutability

After construction:

- FramePacket metadata should not be modified.
- ESP32Metadata should not be modified.
- SyncPair should not be modified.

Modules may compute additional statistics, but should not alter the underlying data.

- eyetracker.csv

- eyetracker_validation.txt

Future tools may additionally generate:

- synchronization plots
- playback diagnostics

FramePacket.image

Always stores the original image received from the ESP32.

No overlays may be written into this image.

Display modules must render overlays on image copies.

# Design Rules

1. Every object has exactly one owner.
2. Every object is constructed exactly once.
3. Downstream modules consume objects without modifying them.
4. Dictionaries are not used for communication between modules.
5. Dataclasses define the public interface of the pipeline.
