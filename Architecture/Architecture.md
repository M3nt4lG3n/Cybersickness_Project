# Eyetrackers Architecture

## Last Edit

Architecture Version: 1.0
Last Updated: 2026-07-11
Status: Active

## Overview

The Eyetrackers subsystem is responsible for acquiring synchronized stereo eye-tracking video from two ESP32-S3 cameras and producing synchronized outputs for recording, display, and analysis.

The system is designed around a streaming pipeline. Each stage has a single responsibility and communicates with the next stage using strongly typed dataclasses rather than anonymous dictionaries.

The ESP32 cameras are considered the authoritative time source. Synchronization is performed using timestamps embedded by the ESP32 firmware rather than timestamps generated by the host computer.

# Design Philosophy

The project follows several architectural principles.

## Single Responsibility

Each module should perform exactly one task.

Examples:

- MJPEG parsing belongs only in `mjpeg.py`
- Synchronization belongs only in `sync.py`
- Display belongs only in `display.py`
- Recording belongs only in `recorder.py`

No module should duplicate another module's responsibility.

---

## Single Source of Truth

Each piece of information should be created once.

Examples:

ESP32 metadata is parsed only by MJPEGStream.

FramePacket objects are created only by MJPEGStream.

SyncPair objects are created only by StereoSynchronizer.

All downstream modules consume these objects without modifying them.

---

## Immutable Pipeline

Data flows in one direction.

```
ESP32
    ↓
MJPEGStream
    ↓
FramePacket
    ↓
FrameBuffer
    ↓
StereoSynchronizer
    ↓
SyncPair
    ↓
Consumers
```

Modules should not modify objects created earlier in the pipeline.

---

## Strong Typing

Communication between modules is performed using dataclasses.

Anonymous dictionaries are no longer used for passing image data between modules.

Example:

Correct:

FramePacket

SyncPair

Incorrect:

```python
{
    "left": ...,
    "right": ...,
    "timestamp": ...
}
```

---

# Timing Philosophy

The ESP32 firmware embeds

- Frame Number
- Unix Timestamp (milliseconds)

into every MJPEG frame header.

These timestamps are considered authoritative.

Host receive timestamps are used only for

- latency estimation
- network diagnostics
- dropped frame analysis

Synchronization is never based solely on host receive time.

# Module Responsibilities

## ESP Firmware

Responsible for

- image capture
- frame numbering
- timestamp generation
- MJPEG streaming

Not responsible for synchronization.

---

## MJPEGStream

Responsible for

- HTTP communication
- multipart MJPEG parsing
- JPEG decoding
- metadata extraction
- creation of FramePacket

Does not perform synchronization.

---

## Camera

Responsible for

- acquisition thread
- connection management
- runtime statistics
- buffering

Camera does not decode JPEG images or synchronize frames.

---

## FrameBuffer

Responsible for

- ordered storage of FramePacket objects
- timestamp-based retrieval
- stale frame removal
- thread-safe buffering

FrameBuffer has no knowledge of stereo synchronization.

---

## StereoSynchronizer

Responsible for

- matching left/right FramePackets
- calculating synchronization error
- producing SyncPair objects

This is the only module that determines whether two frames belong together.

---

## Recorder

Responsible for

- writing synchronized video
- preserving synchronized ordering

Recorder receives only SyncPair objects.

---

## CSV Logger

Responsible for

- writing metadata
- synchronization diagnostics
- latency statistics

CSV Logger receives only SyncPair objects.

---

## Display

Responsible for

- visualization
- optional debug overlays

Display never modifies images used for recording.

---

# Ownership Rules

The following objects have exactly one owner.

| Object | Owner |
|---------|-------|
| ESP32Metadata | MJPEGStream |
| FramePacket | MJPEGStream |
| FrameBuffer | Camera |
| SyncPair | StereoSynchronizer |

No other module should instantiate these objects unless the architecture changes.

---

# Future Expansion

The architecture is intended to support future additions including

- pupil detection
- gaze estimation
- blink detection
- calibration
- eye feature extraction
- live streaming
- Unity integration

These additions should consume the existing pipeline rather than replacing it.

---

# Current Status

The project is actively transitioning from dictionary-based communication to strongly typed dataclasses.

Future development should preserve this architecture unless a documented architectural decision supersedes it.
